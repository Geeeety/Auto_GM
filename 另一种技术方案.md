# Auto_GM 另一种技术方案

> 本文记录了一种不同的技术路线思考：如果抛开 React + FastAPI + TypeScript 的技术栈，仅从"实现个人量化交易系统"这个目标出发，应该怎么做。

---

## 一、核心判断

个人量化系统的瓶颈从来不是"前后端分离"或"微服务"，而是**策略迭代速度**和**数据可信度**。每多一个技术边界（Python ↔ TypeScript、前端 ↔ 后端、mock ↔ 真实），就是一个出 bug、拖慢迭代、需要同步维护的位置。

因此核心原则是：**把所有技术边界砍到最少。**

---

## 二、具体做法

### 1. 只用 Python。连前端都是 Python 渲染。

不做 SPA（单页应用），不做 REST API。使用 FastHTML 或 NiceGUI 这类库直接在 Python 里写界面——页面由服务端渲染，交互通过 WebSocket 或 HTMX 完成。

**好处：**

- 没有 TypeScript 类型和 Pydantic schema 需要同步维护
- 数据直接从数据库到 HTML，中间不经过 JSON 序列化/反序列化
- 改一个策略参数立即在同一个进程中生效，不需要重新部署前端
- 调试时只需要一个 `print`，不需要开浏览器 DevTools 看网络请求

### 2. 数据库先行，Schema 即领域模型。

建一个 SQLite 文件，先定义表：

```
bars、signals、orders、trades、positions、risk_events
```

Pydantic 模型从表结构生成，不手写两套。所有数据落地存储——没有"只在内存里的 mock"。

**核心思想：**
- 回测本质上就是对历史 bars 表跑 SQL 查询加一些 Python 计算
- Dashboard 本质上就是 SELECT 语句的可视化
- 不需要单独的"回测引擎"和"API 层"——它们只是同一份数据的不同读取方式

### 3. Notebook 驱动策略研究，而非先搭"策略实验室"界面。

量化策略的诞生地是 Jupyter Notebook，不是网页表单。流程应该是：

```
Notebook 里加载数据 → 写策略 → 跑回测 → 看收益曲线 → 调参数 → 重复
```

只有当策略在 Notebook 里反复验证、值得持续跟踪之后，才把它注册到系统中定时运行。

Dashboard 不是用来"发明策略"的，是用来**监控已经在 Notebook 里跑通的策略**的。

### 4. CLI 优先于 Dashboard。

一个命令行比一个网页更有用：

```bash
autogm backtest --strategy ma_cross --symbol BTC/USDT --from 2024-01-01
```

**理由：**
- CLI 可以脚本化、可以定时执行、可以 CI
- Dashboard 是 CLI 的只读可视化层，不是替代品
- 命令行天然适合自动化，网页天然适合监控——各司其职

### 5. 事件溯源，不是状态机。

不存"当前持仓是多少"，而是存"某时刻开了仓、某时刻平了仓"。当前状态从事件流投影出来。

**这天然适合量化系统：**
- 回测就是加速重放事件流
- 模拟交易就是实时重放事件流
- 排查"为什么这笔交易没成交"只需要看事件日志，不需要猜测状态变化

---

## 三、推荐技术栈

| 用途 | 技术 | 理由 |
|------|------|------|
| 唯一语言 | Python 3.12+ | 不引入第二语言 |
| 持久化 | SQLite | 零配置、单文件、够用 |
| 分析查询 | DuckDB | 对 Parquet/CSV 的分析查询极快，SQL 兼容 |
| Web UI | FastHTML 或 NiceGUI | Python 原生渲染，WebSocket 推送，零 JavaScript |
| 策略研究 | Jupyter | 交互式探索的标准工具 |
| CLI | Click | Python CLI 框架的事实标准 |
| 数据处理 | Polars | 比 pandas 更快，API 更现代 |
| 数据校验 | Pydantic | 模型校验 |
| 可视化 | Plotly | 交互性（缩放、hover、子图联动）不输 Recharts，且一行 `fig.show()` |
| 测试 | pytest | Python 测试标准 |

## 四、关于"可视化不足"的回应

### 什么会弱一些

如果"可视化"指的是 Recharts 那样的 SVG 交互图表和 CSS 动画过的卡片——是的，单栈方案会弱一些。

### 什么会更强

如果"可视化"指的是**能看清你的策略在做什么**——单栈方案更强：

**数据流完全透明：**

```
单栈方案：[数据库] → [Polars DataFrame] → [Plotly figure] → [浏览器渲染]
React方案：[数据库] → [Pydantic] → [JSON] → [HTTP] → [JSON] → [TypeScript] → [React] → [Recharts] → [SVG]
```

单栈方案的整条链路在同一个 Python 进程里。出问题时只需 `print(df.head())`。React 方案需要在浏览器的 Network 面板、Python 终端、TypeScript 类型文件之间来回跳——每一步都可能出错。

**实时监控不需要"刷新"：**

NiceGUI 基于 WebSocket，策略信号、风控事件、订单状态变化是**推送**到前端的。React 方案要做实时推送，需要额外引入 WebSocket 或 SSE，前后端都要改——又是一个跨边界工程。

**策略与可视化零距离：**

```python
# 在 Notebook 里
df = load_bars("BTC/USDT", "1h", "2024-01-01", "2024-06-01")
signals = ma_cross_strategy(df, fast=12, slow=48)
plot_with_signals(df, signals)  # 一行出图
```

React 方案中这个流程是：改 Python 策略代码 → 启动后端 → 打开浏览器 → 切到策略实验室页面 → 看信号。每次调参要多走三步。

**Plotly 不容小觑：**

Plotly 在量化可视化上的能力不输 Recharts，甚至更强——对数坐标、布林带叠加、成交量子图联动、时间轴缩放——这些在量化分析中非常实用，而在 Plotly 中是内置功能。

## 五、"跨边界税"：你真正花时间的地方

### React + FastAPI 方案的时间分配

```
Week 1-2:  搭前后端骨架
Week 3:    定义 Pydantic schema + TypeScript 类型（两边要保持一致）
Week 4-5:  数据层 + 数据导入
Week 6-7:  策略层 + 策略 API 对接前端
Week 8-9:  回测引擎 + 回测 API + 回测前端展示
Week 10:   前端交互优化（loading 状态、错误处理、响应式）
Week 11-12: 调试跨边界 bug
```

注意：量化核心工作（策略、回测、风控）一直被"同步更新 schema、API、前端组件、TypeScript 类型、mock 数据"打断。这就是跨边界税。

### 单栈方案的时间分配

```
Week 1:    SQLite 建表 + 数据导入 + CLI 骨架 + 第一版 Dashboard 页面
Week 2-3:  数据质量检查 + 可视化验证
Week 4-5:  策略框架 + Notebook 回测 + 信号可视化
Week 6-7:  事件溯源回测引擎 + 指标计算 + 完整回测报告
Week 8:    风控接入 + 回测中验证风控效果
Week 9-10: 模拟交易 + 实时 Dashboard
Week 11:   连续运行稳定性测试
Week 12:   复盘 + 文档
```

**你会在第 4 周就已经在跑完整回测了**，而不是还在写 TypeScript 类型。

## 六、"掌控度"的真正含义

掌控一个量化系统，实际上分三层：

| 层 | 内容 | 单栈方案 | React 方案 |
|---|---|---|---|
| 你在看什么 | K 线图、收益曲线、回撤曲线 | Plotly（可能略逊于 Recharts 的 SVG 精细度） | Recharts（更精致的 SVG 渲染） |
| 数据对不对 | 数据完整性、回测复现性 | Python 直接操作数据，调试零成本 | 需要跨 HTTP 边界追踪数据流转 |
| 系统在干什么 | 策略信号、订单状态、风控拦截 | WebSocket 实时推送 + CLI 即时查询 | 需要实现 WebSocket/SSE（当前只有手动刷新） |

后两层——**真正决定你能否信任这个系统的**——单栈方案明显更强。第一层在交互性上 Plotly 不输 Recharts，在视觉精致程度上略逊。

**结论：总掌控度更高，但视觉精致程度略低。**

## 七、分阶段建议

不是非此即彼的选择，而是**分阶段演进**：

### 阶段 1：单栈跑通全链路（现在的你 → 策略跑通）

用 Python + SQLite + NiceGUI/FastHTML + Plotly。目标是"从数据导入到回测曲线，全链路跑通"。这一阶段不需要任何 JavaScript。

**优势：** 所有精力都在量化逻辑上，不被前端工程分散。

### 阶段 2：前端可以换皮（策略跑通之后）

保留后端的 Python 核心不动，前端用任何你喜欢的框架重写。因为此时：
- 数据模型已在实战中被验证
- API 契约已经被 CLI 和 Notebook 充分使用过
- 策略接口已经稳定

写前端只是"换一张皮"——不会再出现前端需求倒逼后端改结构的情况。

**一个反直觉的事实：** 先写后端 API 再写前端，和先写 CLI 再写前端，前端的开发难度是一样的。但前者的后端被 API 契约束缚住了，后者的后端是自由的。一个自由的、被反复使用的后端，最终产出的 API 设计会比"为了前端而设计的 API"更好。

---

## 八、总结

| 维度 | React + FastAPI 当前方案 | 单栈方案 |
|------|--------------------------|----------|
| 迭代速度 | 慢（跨边界同步） | 快（一个 print 搞定） |
| 视觉精致 | 高（现代 Web 标准） | 中高（Plotly 也很强） |
| 实时监控 | 需额外开发 | 内置 WebSocket 推送 |
| 策略研究 | 需启动全套服务 | Notebook 即环境 |
| 调试难度 | 高（多边界排查） | 低（单进程） |
| 可自动化 | 依赖 API | CLI 天然可脚本化 |
| 简历亮点 | 前后端分离 | 务实工程 |
| 核心目标达成 | 能达成 | 更快达成 |

两种方案没有对错。如果你更看重"学习前后端分离的工程实践"和"产出一个看起来像产品的 Dashboard"，当前方案完全合理。如果你更看重"尽快跑通量化逻辑、尽快验证策略想法"，单栈方案是更短的路。

选择权在你。
