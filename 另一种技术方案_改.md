# Auto_GM 另一种技术方案_改

> 本文是在原《另一种技术方案.md》的基础上，根据新的需求倾向进行修订后的技术路线说明。  
> 核心变化：Auto_GM 不再以“React + FastAPI 前后端分离 Dashboard 产品壳”为主线，而转向“Python-first 的个人量化工程工作台”。

---

## 一、需求变化总结

经过重新判断，项目需求发生了以下变化：

| 维度 | 原先隐含倾向 | 新的明确倾向 | 对技术路线的影响 |
|---|---|---|---|
| 第一阶段速度 | 先搭完整前后端骨架 | Python 栈优先 | 应减少 TypeScript、React、REST API 等跨边界成本 |
| 量化研究 | 通过 Dashboard 暴露策略实验室 | Python / Notebook 优先 | 策略研究应在 Notebook 和 Python 模块中完成 |
| 数据可信度 | mock API 先跑通展示 | 数据库、数据校验、落库优先 | 应先做 SQLite/Parquet/DuckDB 数据层 |
| Dashboard 观感 | 现代 Web Dashboard | 简洁、好用、耐看、可视化交互 | 不必须使用经典前端技术，重点是人机交互效率 |
| 长期产品化 | 可能走前后端分离产品形态 | 暂未确定 | 暂不为未知产品化提前支付复杂度成本 |
| 学习价值 | 前后端工程 + 量化系统 | 更集中在量化工程 | 学习主线应放在数据、策略、回测、风控、事件追踪 |

新的判断是：

**Auto_GM 当前最需要的不是一个“像产品的前端”，而是一个真正可用的个人量化研究、回测、风控和可视化工作台。**

---

## 二、修订后的核心判断

个人量化系统的第一性问题不是前后端工程，而是：

1. 数据是否可信；
2. 策略是否容易迭代；
3. 回测是否可复现；
4. 风控是否能追溯；
5. 系统状态是否能被清楚看见；
6. 研究过程是否能沉淀为可运行模块。

因此，Auto_GM 的第一阶段目标应从：

```text
Dashboard 页面 → API → mock 数据 → 后续接入量化核心
```

调整为：

```text
数据落库 → 数据质量检查 → Notebook 策略研究 → CLI 回测 → 事件日志 → Python 可视化工作台
```

也就是说：

**Dashboard 不再是主系统本身，而是量化核心的可视化交互入口。**

---

## 三、项目新定位

Auto_GM 应重新定位为：

> 一个 Python 驱动的个人量化研究、回测、风控和模拟交易工作台。

它不是一个面向外部用户的金融产品，也不是一个以展示为中心的 Dashboard 项目。

它首先是一个给自己使用的工程系统：

- 可以导入和检查市场数据；
- 可以在 Notebook 中快速研究策略；
- 可以通过 CLI 重复运行回测；
- 可以把回测结果、信号、订单、成交、风控事件全部落库；
- 可以通过简洁耐看的可视化界面观察系统状态；
- 可以在后续需要时再产品化。

---

## 四、修订后的技术原则

### 1. Python-first

项目第一阶段只引入一个主要编程语言：Python。

这样可以避免：

- Python 与 TypeScript 类型同步；
- Pydantic schema 与前端 types 重复维护；
- mock API 与前端 fallback mock 双重维护；
- 后端逻辑、前端状态、浏览器 Network 面板之间来回调试。

Python 应承担：

- 数据导入；
- 数据校验；
- 策略研究；
- 回测；
- 风控；
- 事件记录；
- 可视化交互；
- CLI 自动化。

### 2. 数据库先行

系统应先有可信数据层，再有 Dashboard。

第一批核心表建议为：

```text
bars
data_quality_reports
signals
orders
trades
positions_snapshot
risk_events
backtest_runs
backtest_metrics
system_logs
```

其中：

- `bars` 是所有策略研究和回测的基础；
- `data_quality_reports` 记录数据完整率、缺失、重复、异常；
- `signals` 记录策略信号；
- `orders` 和 `trades` 记录交易过程；
- `positions_snapshot` 保存当前状态投影；
- `risk_events` 保存风控触发记录；
- `backtest_runs` 和 `backtest_metrics` 保存回测结果；
- `system_logs` 保存系统运行过程。

### 3. Notebook 负责研究，CLI 负责自动化

策略的诞生地应是 Notebook，而不是网页表单。

推荐流程：

```text
Notebook 加载数据
→ 编写策略
→ 快速回测
→ 可视化信号和收益曲线
→ 调整参数
→ 策略稳定后注册到系统
→ 使用 CLI 重复运行
→ Dashboard 只负责观察和触发
```

CLI 示例：

```bash
autogm ingest --symbol BTC/USDT --timeframe 1h --source csv
autogm check-data --symbol BTC/USDT --timeframe 1h
autogm backtest --strategy ma_cross --symbol BTC/USDT --from 2024-01-01
autogm report --run-id latest
```

### 4. 可视化交互不等于经典前端

新的需求强调的是：

> 可视化上的简洁、好用、耐看，以及程序与使用者之间的交互。

因此，Dashboard 不一定要使用 React、TypeScript、Vite 这类经典前端技术。

第一阶段推荐：

```text
NiceGUI + Plotly
```

原因：

- 仍然运行在浏览器里；
- 可以保留按钮、表格、参数输入、日志面板、图表等交互；
- 不需要维护前端工程；
- 与 Python 数据结构、回测结果、Notebook 图表天然接近；
- 更适合个人量化工作台。

Plotly 负责：

- K 线；
- 收益曲线；
- 回撤曲线；
- 成交点；
- 策略信号；
- 参数对比；
- 数据质量图表。

NiceGUI 负责：

- 页面布局；
- 参数输入；
- 回测触发；
- 表格展示；
- 日志展示；
- 状态面板；
- 简洁交互。

### 5. 事件日志 + 状态投影，而不是二选一

原方案中提出“事件溯源，不是状态机”。修订后应改为：

> 事件日志负责可追溯，状态投影负责快速查看。

也就是说，系统既要保存事件，也要保存当前状态。

例如：

- 开仓、平仓、风控拦截、订单取消，都进入事件日志；
- 当前现金、持仓、浮盈亏、风险暴露，保存为状态投影；
- Dashboard 读取状态投影；
- 复盘和排错读取事件日志。

这样既能追溯，又能快速展示。

## 五、推荐技术栈

| 用途 | 技术 | 说明 |
|---|---|---|
| 主语言 | Python 3.11+ / 3.12+ | 项目主线只保留 Python |
| 数据库 | SQLite | 单文件、零配置、适合个人系统 |
| 分析查询 | DuckDB | 适合查询 Parquet/CSV 和回测结果分析 |
| 数据文件 | Parquet | 保存历史 K 线和中间结果 |
| 数据处理 | Polars | 高性能 DataFrame，更适合批量数据处理 |
| 数据校验 | Pydantic | 校验配置、策略参数、领域对象 |
| 策略研究 | Jupyter Notebook | 交互式研究环境 |
| CLI | Typer 或 Click | 自动化导入、检查、回测、报告 |
| 可视化工作台 | NiceGUI | Python 内写浏览器交互界面 |
| 图表 | Plotly | K 线、收益、回撤、成交点、信号展示 |
| 测试 | pytest | 核心逻辑测试 |
| 后期 API | FastAPI，可选 | 若未来恢复 React 或产品化，再抽出 API |

## 七、对现有 React + FastAPI 项目的处理

现有项目不建议立刻删除。

建议处理方式：

```text
frontend/  → frontend_legacy/
backend/   → backend_legacy/
```

保留原因：

- 它已经证明 Dashboard 的视觉方向；
- 它可以作为未来产品化参考；
- 它的页面结构可以迁移到 NiceGUI 工作台；
- 它能保留第一周成果。

但从现在开始，不建议继续在 React Dashboard 上增加功能。

尤其不建议继续做：

- 更复杂的前端状态管理；
- 更复杂的 mock 数据；
- 策略实验室表单；
- 回测结果前端交互；
- 风控页面细化；
- 系统日志美化。

这些都应等真实量化核心跑通之后再做。

## 八、第一阶段最小可行版本

新的第一阶段目标：

> 用一周左右跑通真实的个人量化工程闭环。

闭环定义：

```text
真实/半真实 K 线数据
→ 落库
→ 数据质量检查
→ 双均线策略
→ 回测
→ 回测结果落库
→ NiceGUI 页面展示
```

### 第 1 天：建数据层

完成：

- SQLite 建表；
- `bars` 表；
- `backtest_runs` 表；
- `backtest_metrics` 表；
- `trades` 表；
- `risk_events` 表。

### 第 2 天：数据导入

完成：

- CSV 导入；
- 或 ccxt 拉取 BTC/USDT K 线；
- 数据写入 `bars`；
- 保证时间戳唯一。

### 第 3 天：数据质量检查

完成：

- 缺失 K 线检查；
- 重复时间检查；
- 异常价格检查；
- 成交量为 0 检查；
- 生成 `data_quality_reports`。

### 第 4 天：第一条策略

完成：

- 双均线策略；
- 信号生成；
- 信号落库；
- Notebook 中可视化信号。

### 第 5 天：第一版回测引擎

完成：

- 买入/卖出撮合；
- 手续费；
- 简单滑点；
- 持仓计算；
- 交易记录；
- 收益曲线；
- 最大回撤；
- 胜率。

### 第 6 天：CLI

完成：

```bash
autogm ingest
autogm check-data
autogm backtest
autogm report
```

### 第 7 天：NiceGUI 工作台

完成 4 个页面：

```text
数据中心
策略研究
回测报告
系统日志
```

## 九、第一版 Dashboard 设计

第一版 Dashboard 不是产品页面，而是工作台。

### 1. 数据中心

展示：

- 当前已有 symbol；
- 时间周期；
- bars 数量；
- 起止时间；
- 数据完整率；
- 缺失区间；
- 重复数据；
- 异常价格。

交互：

- 选择 symbol；
- 选择 timeframe；
- 触发数据质量检查；
- 查看质量报告。

### 2. 策略研究

展示：

- 已注册策略；
- 策略参数；
- 最近一次信号；
- 最近一次回测结果。

交互：

- 修改参数；
- 启动回测；
- 跳转到回测报告。

### 3. 回测报告

展示：

- 收益曲线；
- 回撤曲线；
- 买卖点；
- 总收益；
- 最大回撤；
- 胜率；
- 交易次数；
- 手续费；
- 策略参数。

交互：

- 选择回测 run；
- 对比不同 run；
- 导出报告。

### 4. 系统日志

展示：

- 数据导入日志；
- 回测日志；
- 策略运行日志；
- 风控事件；
- 异常信息。

交互：

- 按模块过滤；
- 按级别过滤；
- 按时间过滤。

## 十、暂不做的事情

第一阶段不做：

- React 重构；
- 复杂前端动画；
- 用户系统；
- 多账户管理；
- 真实下单；
- 高频交易；
- 云部署；
- 多人协作；
- 复杂权限；
- 微服务；
- WebSocket 实时行情；
- 策略市场；
- 过度美化 Dashboard。

这些需求不是否定，而是延后。

当前优先级是：

```text
可信数据 > 可复现回测 > 策略迭代速度 > 风控追溯 > 简洁可视化
```

## 十一、长期演进方向

当 Python 工作台跑通后，再考虑长期产品化。

可能路线有三种。

### 路线 A：继续 Python 工作台

适合：

- 个人使用；
- 本地运行；
- 研究为主；
- 不追求外部用户。

继续使用：

```text
NiceGUI + Plotly + SQLite + DuckDB + CLI
```

### 路线 B：恢复 React 作为展示层

适合：

- 想做成作品；
- 想展示给别人；
- 想获得更精致的 UI；
- 量化核心已经稳定。

此时 React 只做展示层，不再牵引后端设计。

### 路线 C：抽象为 API 产品

适合：

- 多端访问；
- 云端部署；
- 长期运行；
- 多策略、多账户、多用户。

此时再引入：

```text
FastAPI
PostgreSQL
任务队列
鉴权
前端应用
部署系统
```

但这不是当前阶段。

---

## 十二、最终结论

根据新的需求倾向，Auto_GM 的技术路线应正式调整为：

```text
Python-first Quant Workbench
```

即：

- Python 是唯一主语言；
- 数据库是系统核心；
- Notebook 是策略研究入口；
- CLI 是自动化入口；
- NiceGUI + Plotly 是可视化交互入口；
- React + FastAPI 暂时冻结；
- 产品化暂不提前设计；
- 学习价值集中在量化工程。

一句话总结：

> Auto_GM 不应先成为一个漂亮的 Dashboard，而应先成为一个真正可信、可复现、可交互、可扩展的个人量化工程工作台。
